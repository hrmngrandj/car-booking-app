# ระบบจองรถ (Car Booking System) — เวอร์ชันใช้งานจริง

เว็บไซต์นี้เป็น **แอปสมบูรณ์ในตัวเอง** ไม่ต้องพึ่ง Claude อีกต่อไป มี:
- เว็บเซิร์ฟเวอร์ (`server.js`) เขียนด้วย Node.js ล้วนๆ **ไม่ต้องติดตั้ง library เสริมใดๆ**
- หน้าเว็บ (`public/index.html`) ที่คุยกับเซิร์ฟเวอร์ผ่าน API
- ข้อมูลเก็บเป็นไฟล์ `data/db.json` บนเครื่อง/เซิร์ฟเวอร์ที่รันจริง (ไม่ใช่ในเบราว์เซอร์) จึงบันทึกได้จริงและทุกคนที่เข้าเว็บนี้เห็นข้อมูลชุดเดียวกัน

> ทำไมของเดิมกดเพิ่มข้อมูลแล้วไม่เซฟ? เพราะไฟล์ก่อนหน้าเป็น "อาร์ติแฟกต์" ที่ใช้ระบบเก็บข้อมูลของ Claude.ai ซึ่งทำงานได้เฉพาะตอนเปิดอยู่ในหน้าแชทเท่านั้น พอดาวน์โหลดไฟล์ไปเปิดเอง ระบบเก็บข้อมูลนั้นจะไม่มีอยู่ เวอร์ชันนี้แก้ปัญหานั้นโดยมีเซิร์ฟเวอร์และไฟล์เก็บข้อมูลของตัวเอง จึงใช้งานได้จริงนอก Claude

## ⚡ วิธีรันบนเครื่องตัวเอง (ทดสอบ/ใช้ในออฟฟิศ)

1. ติดตั้ง [Node.js](https://nodejs.org) เวอร์ชัน 14 ขึ้นไป (ถ้ายังไม่มี)
2. เปิด Terminal / Command Prompt ไปที่โฟลเดอร์นี้
3. รันคำสั่ง:
   ```
   node server.js
   ```
4. เปิดเบราว์เซอร์ไปที่ `http://localhost:3000`
5. รหัสผ่านแอดมินเริ่มต้นคือ `admin1234` (แก้ไขได้ตามข้อ "เปลี่ยนรหัสผ่านแอดมิน" ด้านล่าง)

ปิดเซิร์ฟเวอร์ด้วย `Ctrl+C`

## 👥 ให้คนอื่นในออฟฟิศใช้งานผ่านวง LAN เดียวกัน

1. รันเซิร์ฟเวอร์ตามข้อด้านบนบนเครื่องคอมพิวเตอร์เครื่องหนึ่ง (เช่นเครื่องที่เปิดทิ้งไว้ในออฟฟิศ)
2. หา IP address ของเครื่องนั้นในวง LAN เช่น `192.168.1.50`
   - Windows: เปิด cmd แล้วพิมพ์ `ipconfig`
   - Mac/Linux: เปิด terminal แล้วพิมพ์ `ifconfig` หรือ `ip addr`
3. คนอื่นในออฟฟิศเปิดเบราว์เซอร์แล้วพิมพ์ `http://192.168.1.50:3000` (แทนที่ด้วย IP จริง)
4. ทุกคนจะเห็นข้อมูลชุดเดียวกันแบบเรียลไทม์ (หน้าเว็บรีเฟรชข้อมูลอัตโนมัติทุก ~6 วินาที)

> ข้อควรระวัง: เครื่องที่รันเซิร์ฟเวอร์ต้องเปิดอยู่ตลอดเวลาที่ต้องการให้คนอื่นใช้งาน ถ้าปิดเครื่อง/ปิดโปรแกรม เว็บจะเข้าไม่ได้

## ☁️ ใช้งานจริงแบบเข้าถึงได้จากทุกที่ (deploy ขึ้นออนไลน์)

แนะนำให้ deploy ขึ้นบริการที่รองรับ Node.js และมี **persistent disk** (พื้นที่เก็บไฟล์ถาวร ไม่หายเมื่อรีสตาร์ท) เช่น:

- **Render.com** — มี free/paid tier รองรับ Node.js และเพิ่ม "Persistent Disk" ได้ (mount ไปที่โฟลเดอร์ `data/`)
- **Railway.app** — รองรับ volume ถาวรเช่นกัน
- **VPS ส่วนตัว** (เช่น DigitalOcean, Vultr, Linode) — รันคำสั่ง `node server.js` แล้วตั้งให้รันตลอดเวลาด้วย `pm2` หรือ `systemd`

**สำคัญมาก:** ถ้า deploy บนโฮสต์ที่ไฟล์ระบบเป็นแบบ "ชั่วคราว" (ephemeral) เช่น บาง free tier ที่ไม่มี persistent disk ข้อมูลในไฟล์ `data/db.json` จะหายทุกครั้งที่แอปรีสตาร์ท ต้องตรวจสอบกับผู้ให้บริการว่ามี persistent storage ให้ mount ที่โฟลเดอร์ `data/` หรือไม่

ตั้งค่า environment variable ที่โฮสต์ (ถ้ารองรับ):
- `PORT` — พอร์ตที่ให้เว็บรัน (บางโฮสต์กำหนดให้อัตโนมัติ)
- `ADMIN_PASSWORD` — รหัสผ่านแอดมิน (แนะนำให้ตั้งใหม่ ไม่ใช้ค่าเริ่มต้น)

## 🔑 เปลี่ยนรหัสผ่านแอดมิน

วิธีที่ 1 — แก้ไขในไฟล์ `server.js` บรรทัด:
```js
const ADMIN_PASSWORD = process.env.ADMIN_PASSWORD || 'admin1234';
```
เปลี่ยน `'admin1234'` เป็นรหัสผ่านใหม่ที่ต้องการ

วิธีที่ 2 — ตั้ง environment variable ตอนรัน (ไม่ต้องแก้โค้ด):
```
ADMIN_PASSWORD=รหัสลับของฉัน node server.js
```

## 💾 ข้อมูลเก็บที่ไหน / สำรองข้อมูลอย่างไร

ข้อมูลทั้งหมด (รถ, คนขับ, การจอง, ประวัติ) อยู่ในไฟล์เดียว: `data/db.json`
- สำรองข้อมูล: คัดลอกไฟล์นี้เก็บไว้เป็นระยะ
- ย้ายเซิร์ฟเวอร์: คัดลอกไฟล์นี้ไปวางที่เครื่องใหม่ในตำแหน่งเดียวกัน (`data/db.json`)

## ✅ ฟีเจอร์ที่มีอยู่

- จองรถ: กรอกผู้ขอใช้ + เลือกคนขับจากลิสต์ + เลือกทะเบียน + วันที่/เวลา
- ดูตารางจองล่วงหน้า 3 วัน (วันนี้ / พรุ่งนี้ / มะรืนนี้) พร้อมไทม์ไลน์แสดงช่วงเวลาที่รถถูกใช้
- กันจองซ้อน: ทะเบียนเดียวกันเวลาทับกันจองไม่ได้, คนขับคนเดียวกันมีคิวทับเวลากันจองไม่ได้ — **ตรวจสอบที่เซิร์ฟเวอร์** (ไม่ใช่แค่ฝั่งเบราว์เซอร์) จึงแม่นยำแม้มีคนจองพร้อมกันหลายคน
- รองรับภาษาไทย / 中文ทั้งหน้าเว็บ
- แผงแอดมิน (ต้องใส่รหัสผ่านที่เซิร์ฟเวอร์ตรวจสอบจริง ไม่ใช่แค่ซ่อนปุ่มในหน้าเว็บ):
  - เพิ่ม/ลบรถ, เพิ่ม/ลบคนขับ (กันชื่อซ้ำ)
  - ยกเลิกการจอง (ผู้ใช้ทั่วไปทำไม่ได้)
  - ปิดรอบวันนี้ — ย้ายข้อมูลวันนี้เข้าประวัติ แล้วล้างตารางจองวันนี้ให้ว่าง
  - ดูประวัติการจองย้อนหลังตามวันที่

## ⚠️ ข้อจำกัดด้านความปลอดภัยที่ควรทราบ

ระบบนี้เหมาะกับการใช้งานภายในทีม/ออฟฟิศขนาดเล็กถึงกลาง การป้องกันด้วยรหัสผ่านเดียวสำหรับแอดมินเป็นแบบพื้นฐาน หากต้องการเปิดให้เข้าถึงจากอินเทอร์เน็ตสาธารณะ ควรพิจารณาเพิ่มเติม เช่น ใช้ HTTPS (เช่นผ่าน Cloudflare หรือ reverse proxy อย่าง Nginx), จำกัดจำนวนครั้งที่ลองรหัสผ่านผิด (rate limiting), หรือใช้ระบบล็อกอินที่รัดกุมกว่านี้
